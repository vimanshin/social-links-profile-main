# План: Social Links Profile (Frontend Mentor, Newbie)

## Контекст

Это учебный проект Frontend Mentor — карточка профиля со ссылками на соцсети. Цель — **не получить готовый сайт**, а пройти базу фронтенда руками: семантический HTML, CSS-переменные, mobile-first, Flexbox, состояния hover/focus, доступность.

Пользователь — новичок. Код пишет он сам, моя роль — давать теорию, обоснования, проверять результат после каждого шага. Я не пишу код за него.

**Что уже есть в проекте:**
- [index.html](index.html) — пустая болванка с текстом без тегов (нужно превратить в семантическую разметку)
- [style-guide.md](style-guide.md) — цвета (HSL), шрифт Inter (400/600/700), брейкпоинты 375/768/1440
- [assets/fonts/](assets/fonts/) — Inter Variable Font + папка static с обычными `.ttf`
- [assets/images/avatar-jessica.jpeg](assets/images/avatar-jessica.jpeg) — аватар
- [design/](design/) — три JPG: mobile, desktop, active-states
- Figma-макет (mobile/tablet/desktop) по предоставленной ссылке

**Решения, согласованные с пользователем:**
- Шрифт Inter — локально через `@font-face` (без Google Fonts)
- Кнопки соцсетей — `<ul><li><a>` (семантический список ссылок)
- `href="#"` — заглушки
- Один брейкпоинт для desktop, без отдельного tablet

### Шаг 1. Семантическая HTML-разметка карточки

Превращаем текстовую болванку в структуру. Без стилей — только теги.

**Что разметим:**
- Внешний контейнер страницы — `<main>` (главный контент)
- Карточка профиля — `<section class="card">` (или `<article>` — обсудим)
- Аватар — `<img>` с осмысленным `alt` (например, "Jessica Randall")
- Имя — `<h1>`
- Локация — `<p>` (или `<span>`)
- Био — `<p>`
- Группа кнопок — `<ul class="card__links">` с пятью `<li>` внутри, в каждом — `<a href="#">`

**Теория, которую разберём:**
- Что такое семантические теги и почему `<div>` — крайний случай
- Почему ссылка `<a>`, а не `<button>` (внешний переход → ссылка)
- Зачем `<ul>` для группы ссылок (скринридер скажет «список из 5 элементов»)
- Атрибут `alt` — что в него писать (не «фото», а описание)
- Иерархия заголовков — `<h1>` ровно один на страницу

**Лучшие практики:**
- Один `<h1>` на страницу
- Списки — для всего, что воспринимается как список
- `alt=""` (пустой) только если картинка чисто декоративная

**Проверю:** валидатор W3C, Outline в DevTools Accessibility, корректный `lang` у `<html>`.

---

### Шаг 2. CSS-фундамент: reset и переменные

В пустой `style.css` пишем основу до любых компонентных стилей.

**Что сделаем:**
- Минимальный reset (`*, *::before, *::after { box-sizing: border-box; margin: 0; padding: 0; }`)
- `:root` с CSS-переменными из [style-guide.md](style-guide.md):
  - `--color-green`, `--color-white`, `--color-grey-700/800/900`
  - `--font-base: 'Inter', sans-serif;`
  - Пару переменных для размеров (`--radius-card`, `--radius-button`) — заполним конкретными числами из Figma

**Теория:**
- `box-sizing: border-box` — почему «без него больно» (padding не ломает ширину)
- CSS custom properties (`--name` / `var(--name)`) — зачем нужны, как наследуются
- Почему HSL читабельнее HEX

**Лучшие практики:**
- Переменные — в `:root`, чтобы были глобально
- Имена переменных описывают **роль**, не значение (`--color-accent`, а не `--color-yellow-green`)
- Mobile-first: базовые стили вне `@media`, для десктопа — `@media (min-width: ...)`

**Почему так:**
- Mobile-first легче масштабируется и быстрее на мобильных (меньше переопределений)
- Переменные = одно изменение → весь сайт обновился (DRY)

### Шаг 3. Подключение шрифта Inter через @font-face

**Что сделаем:**
- В `style.css` объявить `@font-face` с путём к `assets/fonts/static/` (там обычные `.ttf` для весов 400, 600, 700)
- Указать `font-display: swap` (показывать запасной шрифт, пока грузится Inter)
- Применить `font-family: 'Inter', sans-serif;` к `body`
- Базовый `font-size: 14px` на body (по style-guide)

**Теория:**
- Что такое `@font-face` и три «семьи» fallback (`sans-serif`, `serif`, `monospace`)
- `font-display: swap` vs `block` — UX при медленном соединении
- Variable fonts vs static — почему мы выбрали static (проще для новичка)

**Лучшие практики:**
- Всегда указывать fallback (`'Inter', sans-serif`) — если шрифт не загрузится, текст всё равно читаем
- Подключать только нужные веса (400, 600, 700), а не все 9 — экономим трафик

**Проверю:** в DevTools → Network → Fonts должны грузиться 3 файла; в Computed → font-family указан Inter.

### Шаг 4. Стили страницы и карточки (mobile-first)

**Что сделаем (mobile):**
- `body`: фон `--color-grey-900`, цвет текста `--color-white`, центрирование карточки через flex (`display: flex; justify-content: center; align-items: center; min-height: 100vh;`)
- `.card`: фон `--color-grey-800`, padding, `border-radius`, фиксированная или max-ширина (точное значение возьмём из Figma — около 327px)
- Внутри карточки — вертикальное расположение через `display: flex; flex-direction: column; align-items: center;`

**Теория:**
- Flexbox: основная и поперечная оси, `justify-content` vs `align-items`
- Центрирование «и по горизонтали, и по вертикали» — классическая задача
- `min-height: 100vh` — почему `min-height`, а не `height` (контент может быть выше экрана)
- `rem` vs `px` — почему `rem` гибче (адаптируется к настройкам пользователя)

**Лучшие практики:**
- Размеры — в `rem` (кроме border 1px)
- Не задавать жёсткую `height` контейнерам с контентом
- Карточке давать `max-width`, а не `width` — на узких экранах подстроится

**Почему так:** жёсткие пиксели ломают доступность (пользователь увеличил шрифт в браузере — у тебя сломалась вёрстка).

### Шаг 5. Стили аватара, имени, локации, био

**Что сделаем:**
- Аватар: круг (`border-radius: 50%`), фиксированный размер (~88px по Figma), отступ снизу
- `<h1>`: `font-weight: 600`, размер из Figma (~24px)
- Локация: `color: var(--color-green)`, `font-weight: 700`
- Био: обычный текст `--color-white`, центрирован

**Теория:**
- `border-radius: 50%` для идеального круга — и почему важно, чтобы изображение было квадратным
- `text-align: center` — наследуется детям
- Различие `font-weight` числами (400 = normal, 700 = bold)
- `line-height` без единиц измерения (`line-height: 1.5`) — почему это лучше пикселей

**Лучшие практики:**
- Заголовку всегда `line-height` ~1.1–1.3, тексту ~1.4–1.6
- Отступы между элементами — `margin-bottom` у предыдущего ИЛИ `margin-top` у следующего (выбрать одну стратегию и держаться её). Альтернатива — `gap` у родителя-flex.

**Почему gap — современнее:** не нужно бороться с «схлопыванием margin», работает одинаково для flex и grid.

### Шаг 6. Стили списка кнопок (базовое состояние)

**Что сделаем:**
- Убрать у `<ul>` маркеры (`list-style: none`) и отступы
- `<ul>` сделать flex-колонкой с `gap` между элементами
- `<a>` внутри `<li>`: `display: block` (чтобы кликался весь прямоугольник), фон `--color-grey-700`, белый текст, padding, `border-radius`, `text-align: center`, `text-decoration: none`, `font-weight: 700`

**Теория:**
- Почему `<a>` по умолчанию `inline` и почему `display: block` решает проблему кликабельной области
- Зачем сбрасывать `list-style` и стандартные отступы у `<ul>`
- `gap` для Flexbox — современная альтернатива `margin` между элементами

**Лучшие практики:**
- Вся карточка-ссылка должна быть кликабельна (большая зона клика = доступность, особенно мобайл)
- `text-decoration: none` для кнопок-ссылок, но **только** если визуально понятно, что это интерактив

**Проверю:** в DevTools убедимся, что hit-area = весь прямоугольник, а не только текст.

### Шаг 7. Состояния hover, focus, focus-visible

Это пункт из user stories — обязательный.

**Что сделаем:**
- `:hover` для `<a>`: фон `--color-green`, цвет текста `--color-grey-900` (по `active-states.jpg`)
- `:focus-visible` — то же самое + чёткий outline или другой визуальный маркер
- `transition: background-color 0.2s, color 0.2s` для плавности

**Теория:**
- Разница `:hover`, `:focus`, `:focus-visible`, `:active`
- Почему `:focus-visible` лучше, чем просто `:focus` (не подсвечивается при мышиных кликах, только при Tab)
- Зачем `transition` — снижает «дёрганье» интерфейса

**Лучшие практики:**
- **Никогда** не убирать `outline: none` без замены — это ломает клавиатурную навигацию
- `:hover` и `:focus-visible` обычно стилизуют одинаково, но `:focus-visible` обязателен
- Цвета hover/focus должны иметь достаточный контраст по WCAG (зелёный на тёмном тексте — проверим в DevTools)

**Почему так:** ~15% пользователей хоть раз навигируют клавиатурой (форма, accessibility-режим). Без видимого фокуса они слепнут.

**Проверю:** нажми Tab 6 раз — должен последовательно подсвечиваться каждый из 5 пунктов; mouse-hover — плавный переход.

### Шаг 8. Desktop-стили через @media

В макете отличия минимальны — карточка чуть шире, отступы чуть больше.

**Что сделаем:**
- `@media (min-width: 768px)` или `1024px` — увеличить `max-width` карточки (~384px), padding, font-size заголовка (если меняется в Figma)

**Теория:**
- Синтаксис `@media` и логика «начиная с какой ширины» в mobile-first
- Почему именно `min-width` (а не `max-width`) при mobile-first подходе

**Лучшие практики:**
- Брейкпоинты выбирать **по дизайну** (где макет ломается), а не по устройствам
- Не плодить много брейкпоинтов без нужды — здесь хватит одного
